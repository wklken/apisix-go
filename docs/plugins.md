# APISIX 3.17 Plugin Support

> Current Go-native plugin status, supported behavior, and remaining compatibility boundaries.
>
> Official comparison baseline: Apache APISIX `release/3.17` (104 default plugins). Evidence is maintained against `pkg/plugin/init.go`, the plugin packages, route/proxy integration, focused tests, and the repository verification gate.
>
> Status audit: 2026-07-14.

## Summary

- Official APISIX 3.17 default plugins: **104**
- Registered default plugins: **100** (**96.2%**)
- Supported at the documented Go-native monitoring level: **89** (**85.6%**)
- Deferred because of native/runtime or separate-subsystem boundaries: **9**
- Unsupported or intentionally not required in normal parity: **6**
- Additional documented rows: **14** (12 README/native extras and 2 registered non-default protocol entries)
- Total plugin rows in the status table: **118**

## Integrated plugin tests

The standalone integration suite under [`t/plugin`](../t/plugin/README.md) is
pinned to Apache APISIX commit
`c3d7d5ec69774121f53d2e20d29d09c816795dd7`. Its six manifests map 170
upstream `TEST` blocks to 113 executable standalone runs. Each run writes a
fresh `config.yaml` and `apisix.yaml`, starts the real APISIX-Go CLI, sends the
source-equivalent request, and asserts the response, upstream request, or
startup log.

`referer-restriction.t`, `uri-blocker.t`, and `redirect2.t` currently have no
skipped groups. Three concrete non-standalone groups remain in the earlier
`redirect.t`, `proxy-rewrite.t`, and `response-rewrite.t` manifests: frontend
TLS handshake behavior and two Admin API/etcd serialization checks. Generic
source-enumeration skips are not treated as coverage.

The four intentionally unregistered defaults are `ext-plugin-pre-req`, `ext-plugin-post-req`, `ext-plugin-post-resp`, and `inspect`. They require an external plugin runner or Lua/OpenResty runtime. The bounded `serverless-pre-function` and `serverless-post-function` compatibility implementations remain documented as not-required native/runtime parity.

## Status definitions

- **Supported (monitor):** registered, usable through the Go request/response pipeline, and covered by focused tests for the important paths.
- **Supported (README):** a registered Go implementation with the documented bounded behavior; the row is retained for an extra/native surface outside the 104-default count.
- **Deferred:** usable behavior exists, but completing parity needs a separate stream, transport, health, or runtime contract.
- **Not required (native/runtime):** the remaining behavior is native to NGINX, OpenResty, Lua, or an external plugin runtime and is intentionally outside normal Go-native parity.
- **Remaining jobs:** concrete compatibility boundaries or monitor-only follow-ups. No percentage-only implementation work is implied.

## Cross-plugin remaining work

These items require a new owner or runtime contract and should not be implemented as disconnected plugin-local work:

1. **General stream-plugin context:** define a stream context and lifecycle owner for every supported stream plugin, then add end-to-end fixtures. The current stream design is documented in [`design.md`](design.md).
2. **Active upstream health probes and health-status API:** define scheduler ownership, state persistence, and `/v1/healthcheck` semantics before implementation.
3. **Kafka external-broker smoke coverage:** add only when an external integration environment and credential-safe CI contract are available.
4. **gRPC streaming and cancellation:** expose an explicit route binding before adding a streaming response owner.
5. **Concrete expression, regex, or schema mismatches:** reproduce the APISIX-vs-Go mismatch first, then add the smallest regression and fix.

## Scope boundary

The following remain intentionally outside normal parity work: arbitrary Lua execution, `ngx_lua` APIs, exact OpenResty phase and body-filter timing, NGINX buffering and connection-state internals, OCSP/TLS stapling internals, exact shared-dict/lrucache/batch-manager behavior, OpenResty cosocket pooling, and external plugin-runner protocol compatibility.
