# APISIX 3.17 Plugin Support

> Consolidated plugin status, per-plugin coverage, execution backlog, and remaining gap notes.
>
> Official comparison baseline: Apache APISIX `release/3.17` (104 default plugins). Local implementation evidence comes from `pkg/plugin/init.go`, the registration smoke test, the route/proxy pipeline, and the focused plugin tests.

## Overall comparison

| Measure | Official APISIX 3.17 | Go implementation | Percentage of official inventory |
|---|---:|---:|---:|
| Default plugin inventory | 104 | 104 compared | 100% |
| Registered default plugins | 104 | 100 | 96.2% |
| Supported at the documented Go-native monitoring level | 104 | 89 | 85.6% |
| Deferred: native/runtime or separate subsystem | 104 | 9 | 8.7% |
| Unsupported / not required in normal parity | 104 | 6 | 5.8% |
| Normal parity items marked `implement` | — | 0 | — |

The four missing registrations are `ext-plugin-pre-req`, `ext-plugin-post-req`, `ext-plugin-post-resp`, and `inspect`. The six `not-required-native` entries include those four missing registrations plus the bounded `serverless-pre-function` and `serverless-post-function` compatibility entries. The nine deferred entries are native/runtime or separate-subsystem boundaries, not silently supported implementations.

For this document, `Supported` means the matrix status is `Supported (monitor)`: the plugin is registered, its main Go-native request/response behavior is usable, and focused tests cover the important paths. The percentage is a documented Go-native coverage estimate against the official APISIX 3.17 behavior, not a claim of OpenResty/NGINX/Lua implementation fidelity.

#### Definition of “supported”

For this project, a plugin is supported when all of the following are true:

1. `plugin.New(name)` returns a real implementation or an explicitly documented native-runtime deferment exists.
2. The APISIX 3.17 user-facing schema is accepted for the supported configuration surface.
3. The main request/response behavior works through the Go route and proxy pipeline.
4. Focused tests cover the important success, rejection, and boundary paths.
5. README and this document state the remaining limitations honestly.

Registration alone is not sufficient. Configuration-only or context-propagation stubs must remain marked partial until their runtime behavior is implemented or intentionally classified as native-only.

### Suggested Implementation Order

The OAS validator also accepts free-form exploded `form` query objects backed by `additionalProperties`; duplicate scalar properties remain rejected. It now supports bounded Parameter Object `content` serialization for one JSON/`+json` or `text/plain` media type with schema validation; malformed JSON, duplicate values, and unsupported parameter codecs are rejected explicitly.

1. Keep the shared Go passive health-aware upstream/load-balancer abstraction as the owner for observed HTTP/TCP outcomes; active probes and the `/v1/healthcheck` state API remain explicit deferments rather than plugin-local fakes.
2. Keep the official Kafka PubSub owner and TLS/SASL/PLAIN boundary stable; external broker smoke coverage and mechanisms beyond PLAIN remain later work.
3. Revisit gRPC annotation/hooks/streaming only after the Go route/proxy layer exposes an explicit protobuf streaming binding and cancellation contract.
4. Fix `body-transformer`, `data-mask`, `oas-validator`, `traffic-split`, `response-rewrite`, `fault-injection`, or `uri-blocker` only when a concrete APISIX-vs-Go mismatch is reproduced; the current normal request/validation slices are already covered by focused and route-chain tests.
5. Keep OpenResty/NGINX/Lua-native behavior and the four missing external-runtime defaults documented as deferred rather than adding placeholders.
